## Overview

OmniFlow is a comprehensive web and mobile platform that combines modern travel planning with artificial intelligence. Users enter their destinations, budgets, and travel styles to receive personalized itineraries — complete with flights, hotels, and daily timelines generated by AI.

Unlike static platforms like Booking.com or TripAdvisor, OmniFlow doesn't just list options — it **builds a complete travel plan**, lets users **customize every detail**, and creates a **social community** where travelers share, fork, and improve each other's routes.

> *"Making travel planning accessible, intelligent, and social for everyone."*

The project is developed in two phases as part of the **Computer Engineering** program at **Akdeniz University**:

| Phase | Scope | Duration |
|-------|-------|----------|
| **CSE332 — Software Engineering (MVP)** | Prototype with mock data, core features, weekly demos | ~3 months |
| **Graduation Project** | Full platform with real APIs, mobile app, RAG, AI agents | ~1 year |

## Features

### 🗺️ Trip Planner — 5-Step Smart Itinerary Builder

The core feature. Users are guided through a step-by-step flow to create a personalized travel plan:

1. **Basics** — Destination, dates, travelers, budget tier, travel style
2. **Flights** — 3-4 flight options from the database, user picks one
3. **Hotels** — 3-4 hotel options from the database, user picks one
4. **AI Timeline** — LLM generates a daily schedule from 20-30 filtered places
5. **Customize** — User edits the AI skeleton: add/remove stops, reorder, lock times

**Budget Tiers:** `Economy` · `Standard` · `Premium`

**Travel Styles:** `Solo` · `Family` · `Adventure` · `Luxury` · `Relax`

### 📍 Stop Management

Full control over every point in the itinerary:

- Add places from the catalog or create custom stops (*"Walk along the river"*)
- Set duration, arrival time, and transport mode per stop
- **Time locking** — *"My Colosseum ticket is at 14:00, don't touch it"* — AI respects locked times during replanning
- **Fallback places** — Weather API integration for indoor alternatives
- **Drag-and-drop** reordering via LexoRank algorithm
- Separate activity & transport cost tracking for budget pie charts
- One-tap booking reference copy

### 🍴 Fork System

Inspired by GitHub — see a route you like? Fork it in one click:

- Full copy of the original route under your account
- Add/remove stops, change dates, update traveler count
- Map updates in real-time
- Original creator earns karma when forked
- You cannot fork your own route

### 🔍 Explore — Route Catalog

Discover published routes with rich filtering:

- Trending routes ranked by `popularity_score`
- Filter by city, budget tier, travel style, and tags
- View upvotes, fork count, and view statistics
- Route labels: *budget-friendly*, *luxury*, *adventure*, etc.
- Flag system for incorrect labels + admin correction

### 📱 Community Feed

A social stream separate from Explore, for sharing travel experiences:

| Post Type | Description |
|-----------|-------------|
| **Photo** | Travel photos with location tags |
| **Tip** | General travel advice — *"In Italy, coffee is drunk standing up"* |
| **Route** | Route share — visible in both Feed and Explore |

**Feed Tabs:** `For You` (personalized) · `Following` · `Latest` (chronological)

**Interactions:** Upvote (no downvote) · Comments with 1-level replies · @mentions · User follow system

### 💬 Community Tips

Route-specific tips that don't appear in the main Feed (Foursquare-style):

- General tips for the entire route: *"Check into the Day 3 hotel before 8 PM"*
- Place-specific tips: *"Louvre — buy tickets online, you'll wait 2 hours otherwise"*
- Quality tips surface via upvotes

### ⭐ Karma System

Gamification that rewards quality contributions:

| Action | Points |
|--------|--------|
| Publish a route | +10 |
| Route gets forked | +5 |
| Tip upvoted | +2 |
| Post upvoted | +1 |

**Anti-farming protection:** Same action can only award points once. Toggling publish/unpublish or re-forking won't grant duplicate karma. Reaching **1,000 karma** earns the `Pro Explorer` badge (dynamic, not stored as a role).

### 🤖 AI Fallback System

When the OpenAI service goes down, the system automatically switches to a rule-based engine:

1. Filter places by user's budget and travel style
2. Sort by popularity score
3. Generate timeline skeleton from a fixed template
4. Notify user: *"AI is temporarily unavailable"*

## Architecture

### Tech Stack

| Layer | Technology |
|-------|-----------|
| **Backend** | ASP.NET Core 8.0 |
| **Frontend** | Next.js 14 (React) |
| **Mobile** *(graduation)* | Flutter |
| **Database** | Azure Database for PostgreSQL 15 + PostGIS 3.3 |
| **Cache & Queue** | Redis 7.0 |
| **Maps** | Google Maps SDK |
| **AI / LLM** | OpenAI GPT-4o (prototype) → RAG architecture (graduation) |

### Data Architecture

Three-layer strategy designed for seamless API migration:

```
┌─────────────────────────────────────────────────┐
│              Application Layer                   │
│         (reads only from database)               │
├─────────────────────────────────────────────────┤
│              Database Layer                       │
│        Azure PostgreSQL + PostGIS                 │
├─────────────────────────────────────────────────┤
│            Data Source Layer                      │
│   MVP: LLM-generated mock JSON (10-20 cities)   │
│   Grad: Amadeus API → ETL pipeline → PostgreSQL  │
└─────────────────────────────────────────────────┘
```

The application layer is **data-source agnostic** — switching from mock data to Amadeus API requires only an ETL pipeline update, zero application code changes.

### Database Schema

**Azure Database for PostgreSQL 15 + PostGIS 3.3** — 20 tables with production-grade constraints.

```
┌───────────────────── Core ─────────────────────┐
│  users · trips · places · stops · flights · hotels    │
├───────────────────── Social ───────────────────┤
│  follows · posts · comments · community_tips          │
│  post_upvotes · comment_upvotes · tip_upvotes         │
│  saved_trips · trip_upvotes                           │
├───────────────────── System ───────────────────┤
│  notifications · karma_events · refresh_tokens        │
│  user_blocks · push_tokens                            │
└───────────────────────────────────────────────────────┘
```

#### Key Design Decisions

- **UUID primary keys** — Prevents ID enumeration attacks
- **CITEXT** — Case-insensitive comparisons (`Paris` = `paris`) without `LOWER()` calls
- **Soft delete** — `deleted_at TIMESTAMPTZ` on core tables; data is never physically removed
- **Partial unique indexes** — `WHERE deleted_at IS NULL` allows email/username reuse after soft delete
- **Cache columns** — `karma_score`, `followers_count`, `upvote_count`, `popularity_score` avoid expensive `COUNT(*)` / `SUM()` on every read
- **LexoRank ordering** — `order_index DOUBLE PRECISION` for drag-and-drop without renumbering
- **GEOGRAPHY(POINT, 4326)** — WGS84 coordinate system (same as Google Maps), accounts for earth curvature
- **TIMESTAMP vs TIMESTAMPTZ** — Flights/hotels use timezone-free `TIMESTAMP` (wall clock time); system fields use `TIMESTAMPTZ`
- **Composite FK on comments** — `(parent_comment_id, post_id) → comments(id, post_id)` prevents cross-post reply attacks
- **Karma farming protection** — Unique indexes prevent duplicate points from publish toggling or re-forking
- **CHECK constraints everywhere** — Business rules enforced at the database level: `visited_consistency`, `booked_consistency`, `ai_reasoning_required`, `time_lock_requires_arrival`, `fallback_differs_from_place`, and more

#### Index Strategy

| Type | Usage |
|------|-------|
| **Partial** | `WHERE deleted_at IS NULL` — only active records indexed |
| **GIN** | JSONB fields (`tags`, `photos`, `mentions`, `opening_hours`) and arrays (`budget_tier[]`, `travel_style[]`, `best_months[]`) |
| **GIST** | PostGIS geography columns for spatial queries |
| **Composite** | Frequently co-queried columns (`trip_id, day_number, order_index`) |

## Getting Started

### Prerequisites

- [.NET 8.0 SDK](https://dotnet.microsoft.com/download/dotnet/8.0)
- [Node.js 18+](https://nodejs.org/) & npm/yarn
- [Azure Database for PostgreSQL](https://azure.microsoft.com/en-us/products/postgresql/) — Flexible Server with PostGIS 3.3 and pgvector extensions enabled
- [Redis 7.0](https://redis.io/)
- OpenAI API key

### Installation

```bash
# Clone the repository
git clone https://github.com/your-org/omniflow.git
cd omniflow

# Backend
cd backend
dotnet restore
cp appsettings.Example.json appsettings.Development.json
# Edit appsettings.Development.json with your Azure PostgreSQL connection string & API keys
dotnet ef database update
dotnet run

# Frontend (new terminal)
cd frontend
npm install
cp .env.example .env.local
# Edit .env.local with your API base URL & Google Maps key
npm run dev
```

### Environment Variables

| Variable | Description |
|----------|-------------|
| `DATABASE_URL` | Azure PostgreSQL connection string (`Host=<server>.postgres.database.azure.com;...`) |
| `REDIS_URL` | Redis connection string |
| `OPENAI_API_KEY` | OpenAI API key for GPT-4o |
| `GOOGLE_MAPS_API_KEY` | Google Maps SDK key |
| `JWT_SECRET` | Secret for JWT token signing |

## Security & Data Integrity

- **Soft delete** — User and content data is never physically removed
- **CHECK constraints** — Data integrity enforced at the database level, not just in application code
- **Karma farming protection** — Unique indexes prevent duplicate point exploits
- **Composite primary keys** — Junction tables prevent duplicate relationships
- **ON DELETE RESTRICT** — Critical relationships block accidental data loss
- **Token hashing** — Refresh tokens stored as hashes, never plaintext
- **UUID primary keys** — Prevents sequential ID enumeration
- **Partial unique indexes** — Maintains uniqueness only across active (non-deleted) records

## Performance Optimizations

- **Partial indexes** — `WHERE deleted_at IS NULL AND is_visible = TRUE` keeps index size small
- **GIN indexes** — Fast containment queries on JSONB and array fields
- **GIST indexes** — Efficient geospatial queries via PostGIS
- **Composite indexes** — Frequently co-queried columns grouped together
- **Popularity score cache** — Pre-computed, updated via scheduled job (no per-query aggregation)
- **Karma score cache** — Stored on users table, `karma_events` serves as audit log only
- **Counter caches** — `followers_count`, `fork_count`, `upvote_count` avoid expensive JOINs
